## FastAPI Table API Concept


| Concept       | Frontend behavior                                                                 | FastAPI behavior                                                                            | Response concept                                                      |
| ------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| List data     | Table composable calls `api.list(query)` when page loads or filters change        | Read query params, apply filter/search/sort/pagination in database query                    | Return `{ data: rows, total: count }`                                 |
| Pagination    | Sends `page` as 1-based number and `limit` as page size                           | Convert to `offset = (page - 1) * limit` and `limit = limit`                                | `data` contains current page only; `total` contains all matching rows |
| Sorting       | Sends `sortBy` and `sortOrder` from table header sorting                          | Whitelist sortable columns and apply `ORDER BY`                                             | Rows return in requested order                                        |
| Search        | Sends `search` from table search input                                            | Search important text columns for that resource                                             | Same response shape with filtered `total`                             |
| Date filter   | Header datepicker sends `dateFrom` and `dateTo`                                   | Filter by the table date field, usually `createdAt`, `date`, `registerDate`, or `lastLogin` | Rows match the selected date range                                    |
| Extra filters | Some tables send domain filters like `product`, `action`, `role`, or `categoryId` | Apply exact or `IN` filters in addition to search/date                                      | Rows match all active filters                                         |
| Create        | Form submits a payload to `POST`                                                  | Validate body with Pydantic, insert row, write audit log                                    | Return created row                                                    |
| Update        | Form/slideover submits payload to `PUT` or `PATCH`                                | Validate body, update row, write audit log                                                  | Return updated row                                                    |
| Delete        | Confirm modal calls `DELETE`                                                      | Delete or soft-delete row, write audit log                                                  | Return success object                                                 |
| Export        | Export dialog calls export endpoint when available                                | Reuse same filters, generate CSV or return all export rows                                  | Return file URL or `data` rows                                        |


## Common Query Parameters


| Query param  | Type                  | Used by                              | FastAPI meaning                                  |
| ------------ | --------------------- | ------------------------------------ | ------------------------------------------------ |
| `page`       | `int`                 | All server tables                    | Current page number, 1-based                     |
| `limit`      | `int`                 | All server tables                    | Number of rows per page                          |
| `sortBy`     | `str`                 | Sortable table headers               | Database column or mapped API field to sort by   |
| `sortOrder`  | `asc` or `desc`       | Sortable table headers               | Sort direction                                   |
| `search`     | `str`                 | Search box                           | Global search text                               |
| `dateFrom`   | ISO date/string       | Pages with datepicker                | Start date filter                                |
| `dateTo`     | ISO date/string       | Pages with datepicker                | End date filter                                  |
| `categoryId` | `str`                 | Class/student/category-related lists | Filter rows by category                          |
| `product`    | `str` or comma string | Report table                         | Filter report lines by product/class/course name |
| `action`     | `str` or comma string | History table                        | Filter audit log by action type                  |
| `role`       | `str` or comma string | User/role tables                     | Filter user or role rows                         |

## Table Data Flow


| Step                           | Frontend                                               | FastAPI                                                          | Result in table                        |
| ------------------------------ | ------------------------------------------------------ | ---------------------------------------------------------------- | -------------------------------------- |
| 1. Page opens                  | Composable initializes table state and default sorting | Endpoint receives default `page`, `limit`, `sortBy`, `sortOrder` | First page of rows appears             |
| 2. User searches               | Search ref updates and page index resets to 0          | Endpoint filters text columns by `search`                        | Rows and `total` update                |
| 3. User changes date           | Global date range updates `dateFrom` and `dateTo`      | Endpoint filters date column                                     | Rows match selected range              |
| 4. User changes page size/page | Pagination model changes                               | Endpoint applies offset/limit                                    | New page of rows appears               |
| 5. User sorts column           | Sorting model changes                                  | Endpoint orders by whitelisted field                             | Rows appear in selected order          |
| 6. User creates/edits/deletes  | Form action calls mutation endpoint                    | Endpoint validates, mutates DB, writes audit log                 | Frontend refreshes list endpoint       |
| 7. User exports                | Export dialog calls export endpoint                    | Endpoint reuses filters and builds file/data                     | User downloads or receives export rows |


## FastAPI Implementation Rules


| Rule                                                | Why it matters                                                                                               |
| --------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| Keep list response as `{ data: [], total: number }` | `TableApptable` footer pagination depends on `totalRows`                                                     |
| Keep query param names stable                       | Frontend composables already send `page`, `limit`, `sortBy`, `sortOrder`, `search`, `dateFrom`, and `dateTo` |
| Whitelist `sortBy` fields                           | Prevent invalid SQL/order fields from table header input                                                     |
| Return aliases during migration                     | Existing frontend accepts legacy fields like `Product`, `/products`, `customer`, and `phoneCustomer`         |
| Write audit logs after successful mutations         | History table expects create/update/delete/export events                                                     |
| Do not return raw passwords                         | User table masks password, but backend should store hash and return only masked/empty field                  |
| Preserve hidden Khmer fields                        | `courseNameKm` and `levelKm` are input/saved fields, but current tables display English values only          |
